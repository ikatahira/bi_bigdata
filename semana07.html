Teoria: Schemas de DW

Conteúdo para estudar no seu ritmo, onde e quando quiser

🏗️ Por que precisamos de Schemas específicos para DW?

Em um banco de dados transacional (OLTP), o objetivo é evitar redundância — por isso normalizamos ao máximo. Mas em um Data Warehouse, o objetivo é consulta rápida e análise. Queremos que o analista consiga responder "Quanto vendemos por região, produto e trimestre?" com uma query simples e eficiente.

Para isso, Ralph Kimball (o pai da modelagem dimensional) propôs dois schemas que equilibram desempenho de leitura com clareza de estrutura: o Star Schema e o Snowflake Schema.

💡 Pense assim: no OLTP você otimiza para escrever dados. No DW você otimiza para ler e analisar dados. Isso muda tudo na forma de modelar.

⭐ Star Schema (Esquema Estrela)

O Star Schema é o modelo mais simples e popular de Data Warehouse. Ele tem uma Tabela Fato central rodeada por Tabelas Dimensão desnormalizadas — formando uma estrela quando desenhado.

📊 Tabela Fato

Armazena os eventos mensuráveis: vendas, transações, cliques. Contém métricas numéricas (valores, quantidades) e chaves estrangeiras para as dimensões.

🏷️ Tabelas Dimensão

Descrevem o contexto do fato. Quem comprou? Quando? Onde? Qual produto? São desnormalizadas — todos os atributos ficam em uma única tabela.

🔑 Surrogate Key

Chave substituta gerada artificialmente (id numérico sequencial). Substitui a chave natural do sistema de origem. Garante unicidade e independência.

⚡ Performance

JOINs simples e poucos níveis. O banco de dados faz uma leitura na tabela fato e busca direto nas dimensões. Ideal para ferramentas de BI como Power BI e Tableau.

Exemplo de Star Schema para um sistema de vendas de uma rede de lojas:

-- ── DIMENSÕES (desnormalizadas) ──────────────────────────

CREATE TABLE dim_produto (
  sk_produto       INT PRIMARY KEY,  -- surrogate key
  cd_produto       VARCHAR(20),       -- chave natural
  nm_produto       VARCHAR(100),
  nm_categoria     VARCHAR(50),       -- desnormalizado
  nm_subcategoria  VARCHAR(50),       -- desnormalizado
  nm_marca         VARCHAR(60),
  vl_custo         DECIMAL(10,2)
);

CREATE TABLE dim_cliente (
  sk_cliente       INT PRIMARY KEY,
  cd_cliente       VARCHAR(20),
  nm_cliente       VARCHAR(100),
  nm_cidade        VARCHAR(80),       -- desnormalizado
  nm_estado        CHAR(2),            -- desnormalizado
  nm_regiao        VARCHAR(30),       -- desnormalizado
  ds_segmento      VARCHAR(40)
);

CREATE TABLE dim_tempo (
  sk_tempo         INT PRIMARY KEY,   -- ex: 20240315
  dt_data          DATE,
  nr_dia           INT,
  nr_mes           INT,
  nm_mes           VARCHAR(15),
  nr_trimestre     INT,
  nr_ano           INT,
  fl_fim_semana    BOOLEAN,
  fl_feriado       BOOLEAN
);

-- ── TABELA FATO ──────────────────────────────────────────

CREATE TABLE fato_vendas (
  sk_produto       INT REFERENCES dim_produto,
  sk_cliente       INT REFERENCES dim_cliente,
  sk_tempo         INT REFERENCES dim_tempo,
  qt_itens         INT,
  vl_venda         DECIMAL(12,2),
  vl_desconto      DECIMAL(10,2),
  vl_lucro         DECIMAL(12,2),
  PRIMARY KEY (sk_produto, sk_cliente, sk_tempo)
);

❄️ Snowflake Schema (Esquema Floco de Neve)

O Snowflake Schema é uma extensão do Star Schema onde as Tabelas Dimensão são normalizadas — divididas em sub-dimensões. O resultado visualmente lembra um floco de neve, com as dimensões se ramificando.

📐 Normalização das Dimensões

Ao invés de repetir "Eletrônicos" em mil linhas de produto, criamos uma tabela dim_categoria separada. Reduz redundância, aumenta integridade.

💾 Economia de Espaço

Com dimensões normalizadas, armazenamos atributos repetitivos somente uma vez. Em dimensões muito grandes (ex: produto com 500k itens), a economia pode ser significativa.

🔗 Mais JOINs

A desvantagem: queries precisam de mais JOINs e ficam mais complexas. Ferramentas de BI como Power BI lidam bem, mas SQL manual fica mais trabalhoso.

🛠️ Quando usar?

Snowflake é preferível quando dimensões são muito grandes, mudam com frequência (tipo SCD), ou quando a equipe prioriza consistência dos dados sobre performance.

Veja como a dim_produto do Star vira três tabelas no Snowflake:

-- ── SNOWFLAKE: dimensões normalizadas ────────────────────

CREATE TABLE dim_categoria (
  sk_categoria     INT PRIMARY KEY,
  nm_categoria     VARCHAR(50)
);

CREATE TABLE dim_subcategoria (
  sk_subcategoria  INT PRIMARY KEY,
  nm_subcategoria  VARCHAR(50),
  sk_categoria     INT REFERENCES dim_categoria  -- FK
);

CREATE TABLE dim_produto (
  sk_produto       INT PRIMARY KEY,
  cd_produto       VARCHAR(20),
  nm_produto       VARCHAR(100),
  nm_marca         VARCHAR(60),
  vl_custo         DECIMAL(10,2),
  sk_subcategoria  INT REFERENCES dim_subcategoria -- FK
);

-- A fato_vendas permanece idêntica ao Star Schema.
-- A diferença está apenas nas dimensões normalizadas.

🔍 Conceitos Complementares — Aprofunde-se

🔑 Surrogate Key vs. Natural Key — qual usar? ▼

Natural Key é a chave que já existe no sistema de origem: CPF do cliente, código do produto, número do pedido. Surrogate Key é um ID artificial gerado pelo DW (geralmente INTEGER auto-incremento).

  • Por que preferir Surrogate Key no DW? Porque a chave natural pode mudar no sistema de origem (ex: empresa muda seu código de produto). A surrogate key garante estabilidade histórica.
  • Também permite rastrear múltiplas versões do mesmo registro (SCD Tipo 2), algo impossível com chave natural.
  • Melhora performance: JOINs em INTEGER são mais rápidos que em VARCHAR.
📅 Por que a dim_tempo é especial? ▼

A dimensão tempo é pré-carregada com todos os dias de um período (ex: 2000 a 2035 = ~12.775 linhas). Ela não vem do sistema de origem — é gerada pelo próprio DW.

  • Permite análises como "vendas em fins de semana", "feriados nacionais", "início/fim de trimestre".
  • A surrogate key costuma ser um inteiro no formato YYYYMMDD: 20240315 = 15 de março de 2024.
  • Alguns DWs incluem atributos como semana do ano (ISO), período fiscal da empresa, flag de promoção.
📦 Galaxy Schema — quando ter múltiplas tabelas fato ▼

Em DWs corporativos, é comum ter múltiplas tabelas fato compartilhando dimensões. Esse padrão é chamado de Galaxy Schema (ou Fact Constellation).

  • Exemplo: fato_vendas e fato_estoque compartilham dim_produto e dim_tempo.
  • Dimensões compartilhadas são chamadas de Conformed Dimensions — garantem que "produto" significa a mesma coisa em todos os processos de negócio.
  • É o modelo mais completo mas também o mais complexo de manter.
🎯 Métricas Aditivas, Semi-aditivas e Não-aditivas ▼

Nem toda métrica na tabela fato pode ser somada em todas as dimensões:

  • Aditiva: pode somar em qualquer dimensão. Ex: valor de venda, quantidade vendida.
  • Semi-aditiva: faz sentido somar no tempo mas não sempre. Ex: saldo bancário — soma por produto não faz sentido, mas soma temporal em um ponto sim.
  • Não-aditiva: nunca se soma. Ex: preço unitário, taxa de conversão. Deve-se calcular a média ponderada.

Modelagem de um Star Schema

🔮 +100 XP ao completar

Contexto: Uma rede de farmácias quer analisar suas vendas mensais por produto, loja, farmacêutico e período.

O que fazer:

  1. Identifique as métricas (o que medir): quantidade, valor, desconto...
  2. Identifique as dimensões (contexto): produto, loja, tempo, farmacêutico...
  3. Crie o diagrama do Star Schema com as colunas principais de cada tabela.
  4. Defina as surrogate keys e as chaves estrangeiras da tabela fato.
  5. Escreva ao menos o CREATE TABLE da tabela fato e de duas dimensões.
📤 Entregável: Diagrama (foto/Lucidchart) + SQL no Moodle

Star → Snowflake: Transformação Guiada

🔮 +120 XP ao completar

Contexto: Você recebeu o Star Schema abaixo da atividade 1 de um colega (ou crie o seu próprio).

O que fazer:

  1. Escolha 2 dimensões para normalizar (ex: produto → produto + categoria + subcategoria).
  2. Redesenhe essas dimensões no formato Snowflake, criando as sub-tabelas necessárias.
  3. Atualize o diagrama mostrando o Snowflake resultante.
  4. Escreva uma análise de 3 parágrafos comparando os dois modelos para o cenário da farmácia: quais são as vantagens e desvantagens de cada um neste caso específico?
