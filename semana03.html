Fundamentos de Modelagem Dimensional

Aprenda os blocos fundamentais antes de construir qualquer Data Warehouse

🔍 O que é Modelagem Dimensional?

É uma técnica de design de banco de dados criada por Ralph Kimball especificamente para Data Warehouses. Enquanto sistemas transacionais (OLTP) são otimizados para gravar dados rapidamente, o modelo dimensional é otimizado para consultar e analisar.

Pense assim: um sistema de vendas precisa registrar um pedido em milissegundos. Já um relatório gerencial precisa somar todos os pedidos do trimestre por região, produto e cliente — dois propósitos muito diferentes, dois modelos de dados diferentes.

⚡ Performance

Queries analíticas rápidas com menos JOINs complexos

👁️ Compreensibilidade

Estrutura intuitiva para usuários de negócio, não só técnicos

🔄 Flexibilidade

Fácil adição de novas dimensões sem refatorar tudo

🧭 Navegabilidade

Drill-down e slice-and-dice naturais para análise

Fornecem o contexto aos fatos. Respondem quem, o quê, onde, quando, como e por quê.

Cliente Produto Tempo Loja
Analogia: Se você tem uma venda de R$250,00 (fato), a dimensão dirá que foi o cliente João, comprou o produto X, na loja de SP, no dia 15/01/2025.

⭐ Star Schema (Esquema Estrela)

A estrutura mais comum e recomendada. A tabela fato fica no centro, rodeada pelas dimensões — formando uma estrela.

  • Centro: Tabela fato com as métricas
  • Pontas: Tabelas dimensão diretamente conectadas
  • Dimensões desnormalizadas: Todos os atributos numa única tabela
  • Apenas 1 nível de JOIN: Fato → Dimensão
         DIM_TEMPO
              |
DIM_LOJA -- FATO_VENDAS -- DIM_PRODUTO
              |
         DIM_CLIENTE

-- FATO_VENDAS (centro da estrela)
venda_id    INT   PRIMARY KEY
data_id     INT   FK → DIM_TEMPO
produto_id  INT   FK → DIM_PRODUTO
cliente_id  INT   FK → DIM_CLIENTE
loja_id     INT   FK → DIM_LOJA
quantidade  INT
valor_venda DECIMAL
custo       DECIMAL
lucro       DECIMAL  -- derivado: valor - custo
✅ Menos JOINs ✅ Queries mais rápidas ✅ Fácil entendimento ❌ Mais redundância ❌ Mais espaço em disco

❄️ Snowflake Schema (Esquema Floco de Neve)

Uma variação do Star Schema onde as dimensões são normalizadas — quebradas em sub-dimensões, formando uma estrutura ramificada como um floco de neve.

-- Star Schema (desnormalizado):
DIM_PRODUTO
├── produto_id
├── nome_produto
├── categoria      ← repetido para cada produto
├── subcategoria   ← repetido para cada produto
└── fornecedor     ← repetido para cada produto

-- Snowflake Schema (normalizado):
DIM_PRODUTO        DIM_SUBCATEGORIA  DIM_CATEGORIA
├── produto_id     ├── subcat_id        ├── cat_id
├── nome_produto   ├── nome_subcat      └── nome_cat
└── subcat_id ──→  └── cat_id ──────→

🗂️ Tipos de Tabelas Fato

Nem todos os fatos são iguais — a escolha do tipo correto depende do processo de negócio que você quer modelar.

⚡ Fato Transacional

  • Registra cada evento individualmente
  • Granularidade: nível de transação
  • Volume: altíssimo (milhões de linhas)
  • Totalmente aditiva
  • Exemplos: venda, chamada, transação bancária

📸 Fato Snapshot Periódico

  • Captura estado em intervalos regulares
  • Granularidade: por período (diário/mensal)
  • Volume: moderado e previsível
  • Parcialmente aditiva (não some saldos no tempo)
  • Exemplos: saldo de conta, estoque diário

📋 Fato Snapshot Acumulado

  • Rastreia processos com múltiplos marcos
  • Linha é atualizada conforme avança
  • Múltiplos timestamps (início, marcos, fim)
  • Mede tempo entre etapas
  • Exemplos: pedido (aprovação → envio → entrega)

📐 Tipos de Métricas

✅ Aditivas (70–80%)

Podem ser somadas em todas as dimensões.
Ex: quantidade, receita, custo

SUM(quantidade) por qualquer dimensão = válido

⚠️ Semi-Aditivas

Somáveis em algumas dimensões, mas não no tempo.
Ex: saldo, estoque, headcount

SUM(saldo) por banco = válido · SUM(saldo) no tempo = errado

❌ Não-Aditivas

Jamais podem ser somadas. São derivadas de outras métricas.
Ex: margem %, taxa de conversão

Margem = Lucro / Receita * 100

🎯 Dimensões Especiais que Você Precisa Conhecer

🕐 Dimensão Tempo

A mais universal. Contém dia, mês, trimestre, ano, feriado, dia útil. Toda tabela fato precisa dela.

👻 Dimensão Degenerada

Atributo dimensional que mora na própria tabela fato (sem tabela separada). Ex: número da nota fiscal.

🗑️ Dimensão Junk

Agrupa flags e booleanos que seriam muitos campos soltos. Ex: à_vista/prazo, com/sem desconto, online/presencial.

🤝 Dimensão Conformada

Compartilhada entre múltiplos data marts. A mesma DIM_CLIENTE usada em Vendas e em Suporte.

🎓 Por que Python + SQLite?

SQLite é uma biblioteca SQL embutida — sem servidor, sem instalação. Perfeito para aprender. O Python a acessa com o módulo sqlite3 que já vem instalado. Você vai escrever o mesmo SQL que usaria em PostgreSQL ou SQL Server!

✅ Zero configuração ✅ SQL padrão ✅ Ideal para prototipar DW import sqlite3

🏗️ Desafio 1: Criar as Tabelas do Star Schema

Fácil · +30 XP

Implemente o Star Schema de Vendas. Crie as tabelas dimensão (Tempo, Produto, Cliente, Loja) e a tabela fato (Vendas) com as chaves corretas.

Estrutura esperada:

• DIM_TEMPO: data_id, data_completa, dia, mes, ano, trimestre, dia_semana, feriado

• DIM_PRODUTO: produto_id, nome_produto, categoria, subcategoria, preco_custo

• DIM_CLIENTE: cliente_id, nome, cidade, estado, segmento

• DIM_LOJA: loja_id, nome_loja, cidade, regiao

• FATO_VENDAS: venda_id, data_id(FK), produto_id(FK), cliente_id(FK), loja_id(FK), quantidade, valor_venda, custo, lucro

💡 Dica: Use INTEGER PRIMARY KEY AUTOINCREMENT para surrogate keys. As foreign keys na FATO_VENDAS referenciam as PKs de cada dimensão com FOREIGN KEY (coluna) REFERENCES tabela(pk).
# ✅ SOLUÇÃO — Desafio 1: Star Schema Completo import sqlite3 conn = sqlite3.connect(':memory:') cursor = conn.cursor() # Ativar suporte a chaves estrangeiras no SQLite cursor.execute("PRAGMA foreign_keys = ON") # --- DIMENSÃO TEMPO --- cursor.execute(''' CREATE TABLE DIM_TEMPO ( data_id INTEGER PRIMARY KEY AUTOINCREMENT, data_completa DATE NOT NULL, dia INTEGER NOT NULL, mes INTEGER NOT NULL, ano INTEGER NOT NULL, trimestre INTEGER NOT NULL, dia_semana TEXT NOT NULL, feriado TEXT DEFAULT 'N' CHECK(feriado IN ('S','N')) ) ''') # --- DIMENSÃO PRODUTO --- cursor.execute(''' CREATE TABLE DIM_PRODUTO ( produto_id INTEGER PRIMARY KEY AUTOINCREMENT, nome_produto TEXT NOT NULL, categoria TEXT NOT NULL, subcategoria TEXT, preco_custo DECIMAL(10,2) NOT NULL ) ''') # --- DIMENSÃO CLIENTE --- cursor.execute(''' CREATE TABLE DIM_CLIENTE ( cliente_id INTEGER PRIMARY KEY AUTOINCREMENT, nome TEXT NOT NULL, cidade TEXT, estado TEXT, segmento TEXT DEFAULT 'Pessoa Física' ) ''') # --- DIMENSÃO LOJA --- cursor.execute(''' CREATE TABLE DIM_LOJA ( loja_id INTEGER PRIMARY KEY AUTOINCREMENT, nome_loja TEXT NOT NULL, cidade TEXT NOT NULL, regiao TEXT NOT NULL ) ''') # --- TABELA FATO --- cursor.execute(''' CREATE TABLE FATO_VENDAS ( venda_id INTEGER PRIMARY KEY AUTOINCREMENT, data_id INTEGER NOT NULL, produto_id INTEGER NOT NULL, cliente_id INTEGER NOT NULL, loja_id INTEGER NOT NULL, quantidade INTEGER NOT NULL, valor_venda DECIMAL(10,2) NOT NULL, custo DECIMAL(10,2) NOT NULL, lucro DECIMAL(10,2) GENERATED ALWAYS AS (valor_venda - custo) STORED, FOREIGN KEY (data_id) REFERENCES DIM_TEMPO(data_id), FOREIGN KEY (produto_id) REFERENCES DIM_PRODUTO(produto_id), FOREIGN KEY (cliente_id) REFERENCES DIM_CLIENTE(cliente_id), FOREIGN KEY (loja_id) REFERENCES DIM_LOJA(loja_id) ) ''') conn.commit() print("✅ Star Schema criado com sucesso!") # Verificar tabelas criadas cursor.execute("SELECT name FROM sqlite_master WHERE type='table'") tabelas = cursor.fetchall() for t in tabelas: print(f" 📊 {t[0]}") conn.close()

🔎 Desafio 3: Queries Analíticas (O Poder do DW)

Difícil · +50 XP

Agora é hora de ver o modelo dimensional em ação! Escreva queries SQL que cruzam a tabela fato com as dimensões para responder perguntas de negócio. Aqui você entende por que esse modelo existe.

Escreva queries para responder:

1. 💰 Qual a receita total e lucro total por categoria de produto?

2. 📅 Qual o total de vendas por mês e trimestre?

3. 🏙️ Qual a receita por cidade do cliente e segmento?

4. 🏆 Quais os 3 produtos mais vendidos (em valor)?

💡 O padrão de query dimensional é sempre: SELECT dimensão.atributo, SUM(fato.métrica) FROM FATO JOIN DIM ON fato.fk = dim.pk GROUP BY dimensão.atributo. Adicione múltiplos JOINs para cruzar várias dimensões ao mesmo tempo!
# ✅ SOLUÇÃO — Desafio 3: Queries Analíticas print("="*50) print("📊 QUERY 1: Receita e Lucro por Categoria") print("="*50) cursor.execute(''' SELECT p.categoria, SUM(f.valor_venda) AS receita_total, SUM(f.lucro) AS lucro_total, ROUND(SUM(f.lucro) * 100.0 / SUM(f.valor_venda), 1) AS margem_pct FROM FATO_VENDAS f JOIN DIM_PRODUTO p ON f.produto_id = p.produto_id GROUP BY p.categoria ORDER BY receita_total DESC ''') for row in cursor.fetchall(): print(f" {row[0]:15} | R$ {row[1]:8.2f} | Lucro: R$ {row[2]:7.2f} | Margem: {row[3]}%") print("\n" + "="*50) print("📅 QUERY 2: Vendas por Mês") print("="*50) cursor.execute(''' SELECT t.ano, t.mes, t.trimestre, COUNT(f.venda_id) AS num_vendas, SUM(f.valor_venda) AS receita FROM FATO_VENDAS f JOIN DIM_TEMPO t ON f.data_id = t.data_id GROUP BY t.ano, t.trimestre, t.mes ORDER BY t.ano, t.mes ''') for row in cursor.fetchall(): print(f" {row[0]}/{row[1]:02d} (Q{row[2]}) | {row[3]} vendas | R$ {row[4]:.2f}") print("\n" + "="*50) print("🏙️ QUERY 3: Receita por Cidade e Segmento") print("="*50) cursor.execute(''' SELECT c.cidade, c.segmento, SUM(f.valor_venda) AS receita FROM FATO_VENDAS f JOIN DIM_CLIENTE c ON f.cliente_id = c.cliente_id GROUP BY c.cidade, c.segmento ORDER BY receita DESC ''') for row in cursor.fetchall(): print(f" {row[0]:12} | {row[1]:15} | R$ {row[2]:.2f}") print("\n" + "="*50) print("🏆 QUERY 4: Top 3 Produtos por Receita") print("="*50) cursor.execute(''' SELECT p.nome_produto, p.categoria, SUM(f.quantidade) AS unidades, SUM(f.valor_venda) AS receita FROM FATO_VENDAS f JOIN DIM_PRODUTO p ON f.produto_id = p.produto_id GROUP BY p.produto_id, p.nome_produto, p.categoria ORDER BY receita DESC LIMIT 3 ''') for i, row in enumerate(cursor.fetchall(), 1): print(f" #{i} {row[0]:15} | {row[1]:12} | {row[2]} un. | R$ {row[3]:.2f}") conn.close()
